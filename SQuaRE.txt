ISO/IEC 25000シリーズ SQuaREについて

　SQuaREについて調査しました。文書の量が多いので、隅々まで読んでいるわけではないのですが、
一旦報告します。

●「SQuaREに準拠した品質を満たしていますと対外的に宣言する」には？
　ISO/IEC 25051 (JIS X 25051)に定められた要求事項を満足するソフトウェアであることを保証しなければ
いけません。この企画は商用ソフトウェアが利用者に対して提供する機能が宣言通り動作する信頼を利用
者に与えることを目的としているので、仕様書のような中間生成物に対する活動や生産プロセスに対して
規定はしていません。要するに、
・ISO/IEC 25051に記載された品質要求事項を満たすこと
・上記品質事項を満たしていることを示す試験結果を示すこと
・適切な利用マニュアル等が用意されていること
がこの規格で要求されます。第三者認証は一般社団方針コンピュータソフトウェア境界によるPSQ(パッケージ
ソフトウェア品質認証制度)で取得することができます。
　なお、品質要求事項は品質特性(JIS X 0152:1995または最新のものだとおそらくSQuaREで定めらたもの)に
従い分類されます。

http://kikakurui.com/x25/X25051-2011-01.html
　
● ISO/IEC 25051におけるセキュリティに関する機能要求および、試験項目は？
　結論から言うと現状わかりません。というのも、Webで見つけたJIS X 25051:2011は、SQuaREの前進である
JIS X 0152:1995に基づいています。JIS X 0152:1995でも機能性の副特性としてセキュリティに関する特性
はありますが、ISO/IEC　25051:2011では副特性まで踏み込んで記述されていません。
おそらく最新版のJIS X 25051:2016ではセキュリティに関する特性に触れられているのかもしれませんが
以下で購入しないと読めません。

https://webdesk.jsa.or.jp/


以下メモです。長いので参考程度に。

1. SQuaREとは

　System and software Quality Requirement and Evaluationの略であり、システムまたはソフトウェア
に対する、「品質定義」「品質評価視点」の枠組みを定めている。規格の遍歴としてはISO/IEC 9126から
の流れを汲み、ISO/IEC 9126で定義していたソフトウェア品質特性を時流に合わせて拡張している。
　SQuaREは、品質に対する考え方に特徴があり、品質の観点の広がりをカバーする形で品質を定義する。
SQuaREにおける品質の定義は「明示された状況下で使用されたとき、明示的なニーズおよび暗黙のニーズ
をソフトウェア製品が満足させる度合い」とされている。

2. 品質特性

2.1. 品質の分類

　SQuaRE(ISO/IEC 2501n)では品質を以下のように分類している。
	・製品品質
	・データ品質
	・利用時の品質

2.2. 品質特性・品質副特性の定義

　SQuaRE(ISO/IEC 2501n)では上記の品質特性・品質副特性を分類している。例えば、ISO/IEC 9126から
新たに追加された「セキュリティ」という品質特性は、「製品品質」に属し、以下の品質副特性を持つ。
	[品質特性]：セキュリティ
	[品質副特性]
	・機密性：
		製品又はシステムが、アクセスすることを認められたデータにだけアクセス
		することができることを確実にする度合い。
	・インテグリティ：
		コンピュータプログラムまたはデータに権限を持たないでアクセスすること
		を、システム、製品又は構成要素が防止する度合い。
	・否認防止性：
		事象又は行為が後になって否認されることがないように、行為又は事象が引き
		起こされたことを証明することができる度合い。
	・責任追跡性：
		実態の行為がその実態に一位的に追跡可能である度合い。
	・真正性：
		ある主体又は資源の同一性が主張した通りであることを証明できる度合い。

http://kikakurui.com/x25/X25010-2013-01.html

3. SQuaREの利用

3.1 ISO/IEC 2503n

　SQuaREの利用者は、ISO/IEC 2501nが定める品質特性・品質副特性の観点から、品質要求事項を策定する。
ISO/IEC 2503nでは上記の具体的なプロセスが規定されている。ISO/IEC 2503nのプロセスに従うことで
SQuaREの利用者は開発対象に対する品質要求事項を定義することができる。

http://kikakurui.com/x25/X25030-2012-01.html

3.2 ISO/IEC 2502n

　SQuaREの利用者は、ISO/IEC 2502nが定めるプロセスに従い、品質要求事項の評価基準の設定と測定方法
を定義する。ISO/IEC 2502nが定めるプロセスでは、品質要求事項を評価するための測定量を定め、品質
要求事項をどのように定量的に表すかを決める。加えて測定量の測定方法を定義することも本プロセスで
求められている。測定量には例えばMTBF等があげられる。

http://kikakurui.com/x25/X25021-2014-01.html
	
3.3 ISO/IEC 2504n

　SQuaREの利用者は、ISO/IEC 2504nが定めるプロセスに従い、ISO/IEC 2502nで定めた測定量・測定方法
に従った測定結果を評価する。

http://kikakurui.com/x25/X25040-2014-01.html

4.何をもって準拠していると考えるのか

4.1 ISO/IEC 2505n〜2509n

　ISO/IEC 2505n〜2509nは、システム/ソフトウェア製品一般ではなく、特定の応用範囲や特定のソフト
ウェアの種別についての製品品質を扱う規格類を扱う計画となっている。現在帰省ソフトウェア製品向け
の開発が進んでいる。

4.2 JIS X 25051:2011 (JIS X 25051:2016)

  商用既成(COTS)ソフトウェア製品に対する品質要求事項及び、試験に対する指示。
品質要求事項と試験指示が定まっていることから、2505n〜2509nは2503n->2502n->2504nのプロセス要求に
従ったプロセスに従い、特定の製品に対する品質要求事項と試験に対する指示を定めるものと考えられる。
　T-AGLがCOTSソフトウェアに含まれるかは1の適用範囲を見ると「商用既製ソフトウェア製品の例として
は，次のものがあるが，これらに限定するものではない」とあり、リアルタイムOSもCOTSに含まれている
ことからCOTSに含まれるという解釈は可能だと考えられる。

4.2.1 JIS X 25051:2011が規定すること

- COTSソフトウェア製品の品質要求事項
- COTSソフトウェア製品の試験のための試験部署の要求事項(試験要求・試験項目・試験報告)
- COTSソフトウェア製品適合性評価への指示
　※この企画はCOTSソフトウェアが利用者に対して提供する機能通りに動作する信頼を利用者に与える
　ことを目的としているので、仕様書のような中間生成物に対する活動や生産プロセスに対して規定は
　していない。

4.2.2 JIS X 25051:2011が定める品質要求事項

　以下のような構成となっている
- 製品説明に対する要求事項
- 利用者用文書に対する要求事項
- ソフトウェアに対する要求事項
　機能性・信頼性・使用性・効率性・保守性・移植性・利用時の品質特性に基づき記載されている。
　セキュリティに関する特性が記載されていないのは、SQuaRE制定前(25051が発行されたのが2011年に
　対して2013年に25010が発行されている)に発行された規格なのでJIS X 0152:1995に基づいているため。

4.3 認証

　JIS X 25051:2011 JIS X 25051:2016に基づく第三者認証機関として一般社団方針コンピュータソフトウェ
ア境界によるPSQ(パッケージソフトウェア品質認証制度)がある。

http://www.csaj.jp/psq/
